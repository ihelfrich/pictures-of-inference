# Setup Instructions

This is a one-time guide for getting the repo running on your local machine. After this, the day-to-day workflow is just `make pdf`.

## What you need installed

- **Python 3.9+** with `matplotlib`, `numpy`, `scipy`, `pypdf`. If you have any modern scientific Python distribution, you have these already.
- **A LaTeX distribution.** TeX Live, MacTeX, or MiKTeX. The standard packages are needed: `mathpazo`, `tcolorbox`, `listings`, `microtype`, `hyperref`, `tikz`, `geometry`, `fancyhdr`, `titlesec`. All come with a full TeX Live install.
- **Git.** For version control.
- **Claude Code.** For the actual writing.

## One-time setup

### 1. Get the repo onto your machine

Drop the unpacked starter directory wherever you want it to live. Standard place is `~/Documents/poi/` or `~/code/poi/`. Then:

```bash
cd path/to/poi
git init
git add .
git commit -m "Initial commit: project scaffolding"
```

### 2. Create a private GitHub repo

On GitHub, create a new private repo named `pictures-of-inference` (or whatever you prefer). Add Liz as a collaborator. Then locally:

```bash
git remote add origin git@github.com:YOURNAME/pictures-of-inference.git
git push -u origin main
```

### 3. Verify the build works

From the repo root:

```bash
# Regenerate figures (should produce 2 PDFs in pictures/figures/output/)
make figures

# Build the full book (should produce build/pdf/main.pdf, ~6 pages)
make pdf

# Style check (should print "Clean.")
make check

# Build a single chapter
make ch01
```

If all four commands succeed, the system works. If any fail, fix that before writing anything.

### 4. Verify your matplotlib has Palatino fallback

Run:

```bash
cd pictures/figures && python3 generate_all.py
```

You'll see warnings about Palatino if matplotlib can't find it. That's fine; it falls back to DejaVu Serif and the figures still look good. To get true Palatino in figures, install the font system-wide. Not urgent.

## Daily workflow

### Open a Claude Code session in the repo

```bash
cd path/to/poi
claude
```

Claude Code will read `.claude/CLAUDE.md` automatically. It knows the conventions, the style guide, the structure.

### Drafting a chapter

Tell Claude something like:

> Read docs/voice/ first. Then draft Chapter 1 (Numbers on the page) in `prose/part1/01-numbers-on-the-page.tex`. Use `drafts/v3-chapters/whatisdata.tex` as a content reference but don't copy the voice. Generate figures in `pictures/figures/ch01.py`.

Claude will:
1. Read voice samples.
2. Read existing draft for content ideas.
3. Generate figures.
4. Write the chapter.
5. Run `make pdf` to verify.
6. Run `make check` to verify no banned phrases.

### Revising prose

> Edit the second paragraph of Chapter 1 to feel more like a Feynman opening — build from a small concrete observation rather than starting with "data is..."

Targeted edits, fast iteration.

### Updating figures

> The figure of the data rectangle is too cramped. Move the labels further out and use FIG_FULL size instead of FIG_SINGLE.

Claude edits `pictures/figures/ch01.py` and re-runs.

## Things to put in the repo soon

To unblock real chapter writing, the following should land:

- **Voice samples in `docs/voice/`.** See the README in that folder.
- **A trade panel slice in `data/trade/`.** Even just five countries and ten years to start.
- **A real dataset README in `data/trade/README.md`** documenting source and provenance.

Until voice samples are in place, drafted chapters will be approximations. The voice samples are by far the highest-leverage thing you can provide.

## Things to keep out of the repo

- **Large raw data files.** Use `data/.local/` (gitignored) for anything over 10 MB.
- **Licensed datasets you can't redistribute.** Same — keep them in `.local/` and document them in `data/sources.md`.
- **Generated build artifacts.** `build/` is gitignored. Compiled PDFs are recreated from sources on demand.
- **The figure output folder.** `pictures/figures/output/` is gitignored. Figures are regenerated by scripts.

## When you get stuck

The build system is small and inspectable. If something breaks:

- Look at `prose/main.log` for LaTeX errors.
- Run figure scripts directly with `python3 pictures/figures/ch01.py` to see Python errors.
- Run `make check` to confirm prose passes the style filter.

If Claude does something you don't want, the rule of thumb is: explicitly say what you want different. Claude works best with concrete, specific direction.

## Calling Liz in

Once the repo is on GitHub and she's a collaborator:

1. She clones the repo locally.
2. She runs the same setup.
3. She adds her voice samples to `docs/voice/liz/`.
4. She and you agree on what chapters she'll lead.
5. You both work on branches, merge via pull request.

Before any of that, the two of you should have one conversation about division of labor and what she actually wants this to be. The README and proposal name her as co-author; the practical division comes from talking.
